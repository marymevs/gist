## Problem Statement

Information overload at the start of the day destroys people's morning state. The current default is screens: email, Slack, news, calendar notifications. By the time someone has processed their inbox, they're already reactive. For people with ADHD or screen fatigue, this isn't just unpleasant — it's cognitively harmful.

The opportunity: give someone the minimum viable context to start their day well, delivered on paper — a format that changes HOW they process information, not just WHERE they read it. Paper forces batch processing, removes notifications, and creates a physical boundary between "information time" and "action time."

The full vision: a two-way paper interface for AI agents. Gist arrives on paper. User marks it up. Faxes it back. AI agents execute the marked-up instructions — all without ever touching a screen.

## Demand Evidence

Four distinct people have expressed specific interest in paper delivery:

1. **Bar owner (most acute):** Gets overwhelmed by screens. Specifically said paper would help him stay off his phone, which would help his ADHD. Manages operational inbound from multiple channels (suppliers, staff, reservations). Wants to feel calm before his workday starts.

2. **Quantum engineer:** Spends all day on screens. Wants a means of staying connected to the world that doesn't require them. Sees Gist as permission to disengage.

3. **Director friend:** Thinks better on paper than on screens. Specifically loves the idea of writing on paper rather than typing. Wants the tactile, pen-in-hand experience.

4. **Professor friend:** Assigns papers by hand. Sees potential for plugging paper-based information delivery into the school system. Has offered to help make that happen — a potential distribution channel.

**Demand gap:** All four expressed verbal enthusiasm. None have used the product yet. Zero people have received a fax from Gist. Behavioral demand (actual usage, payment) is unvalidated.

## Premises

1. **Paper's value is cognitive, not nostalgic.** For people with ADHD or screen fatigue, paper changes HOW they process information. The bar owner's ADHD comment is evidence of a medical/cognitive need.

2. **The fax-back loop is the real product.** Without the return path, Gist on paper is a printout — nice but not a platform. The two-way paper interface is what makes this a "life management OS" rather than a "morning newsletter on fax."

3. **Fax is the right delivery mechanism.** Instant, automatic, no user action needed. Phaxio handles the hard parts. "Retro" framing is actually the simplest path to "paper arrives automatically."

4. **The platform vision (north star, not current scope).** Paper as an interface for managing your digital life — email replies, calendar changes, task management, agent-driven actions. The briefing is the first surface. This is aspirational direction, not an engineering requirement for current tiers.

5. **Validate delivery before building the loop.** The biggest risk is not technical — it's that you build the fax-back loop before confirming that one-way delivery is valuable enough to change someone's morning. Sequence: deliver first, observe, then design the return path based on real usage.

## Approaches Considered

### Approach A: Validate-First Fax Delivery
Ship one-way fax delivery to the bar owner within 1 week. No return path. Watch him use it. Design the return path based on what he actually does with the paper.
- **Effort:** S (human: ~3 days / CC: ~2 hours)
- **Risk:** Low
- **Pros:** Fastest path to real user data
- **Cons:** Product may feel underwhelming without the return path. Completeness: 5/10.

### Approach B: Full Two-Way Paper Loop
Build fax-out AND fax-back with Phaxio webhook receive + OCR + basic action parsing. Ship the complete vision.
- **Effort:** L (human: ~3 weeks / CC: ~4-6 hours)
- **Risk:** High (OCR accuracy, action parsing, webhook complexity)
- **Pros:** Ships the actual differentiator
- **Cons:** Long build before any user validation. Completeness: 9/10 but unvalidated.

### Approach C: Tiered Launch (Chosen)
Two pricing tiers designed in parallel, built sequentially. Tier 1 ($8/mo): morning briefing on fax. Tier 2 ($25/mo): full return path with AI agent execution. Design both now, build Tier 1 first, validate, then build Tier 2.
- **Effort:** M for Tier 1, then L for Tier 2
- **Risk:** Medium (staged risk)
- **Pros:** Gets paper in real hands fast while keeping the full vision alive
- **Cons:** Risk of getting comfortable at Tier 1. Completeness: 8/10.

## Recommended Approach

**Approach C: Tiered Launch.** Design the full paper-first OS vision now. Build Tier 1 (one-way fax delivery) first — the pipeline already exists from PR #18, so the remaining work is: domain setup, Stripe billing gate, onboarding flow for external users, and getting a fax to the bar owner. Build Tier 2 (fax-back loop with AI agents) after validating that Tier 1 changes someone's morning.

### Tier 1: Morning Briefing on Paper ($8/mo)
- Daily fax at user's configured delivery time (valid range 5am-9am local time, default 6am). Scheduler stores delivery time in UTC, converted from user's local timezone at preference save. UI displays local time only.
- Content blocks: calendar highlights, email action cards (Action/WaitingOn/FYI categories), world headlines, weather
  - **Graceful degradation:** If Gmail has zero actionable emails, the email section shows "No action emails today." If calendar is empty, shows "No events today — open day." Fax is always sent; sections degrade but never disappear.
- Delivery status tracking (queued/delivered/failed)
  - **Failure handling:** On Phaxio delivery failure, retry once after 5 minutes. If retry fails, mark as `failed` in delivery log and send an email notification to the user (via Resend) saying "Your morning Gist fax failed to deliver — view it on the web at mygist.app/today."
  - **Payment failure notification (screenless-safe):** When the Stripe gate blocks a fax due to past-due payment, send BOTH: (1) a one-page fax saying "Your Gist is paused — update your payment at mygist.app/account to resume delivery" and (2) an email with the same message. The fax ensures screenless users are reached on their primary channel. The email is a fallback. Use the same `faxTemplate.ts` patterns for the payment notification fax (Georgia serif, branded header).
- Print-optimized template (already built: `faxTemplate.ts`)
- **Onboarding fax number validation:** User provides fax number in E.164 format during onboarding. Number is validated via a Phaxio test fax send (`POST /faxes` with a one-page welcome document) to verify the number is reachable. Onboarding blocks if validation fails.
- **Founder billing:** The founder's own account bypasses Stripe for now (self-validation). External users require Stripe from day one.
- **Billing:** Stripe subscription required. No free tier for fax delivery (validates willingness to pay). 7-day free trial, then $8/mo. On payment failure: 3-day grace period, then fax delivery pauses. User receives email notification at each stage. Tier 1 and Tier 2 are separate Stripe products (upgrade path via subscription change).

### Tier 2: Paper Command Interface ($25/mo) — design now, build after validation
- Fax-back: user marks up the briefing and faxes it back to a Gist number
- Phaxio inbound webhook receives the faxed-back document
- OCR + AI parsing: identify markups (circles, checkmarks, crossed-out items, handwritten notes)
- Action mapping: each markup type triggers an AI agent action. Markups target the specific content block on the fax (e.g., the email subject line row, the calendar event row).
  - Circle an email subject line → draft a reply (low-risk: draft is held, not sent)
  - Cross out a calendar event → decline it (**high-risk: destructive action, requires confirmation**)
  - Check a task → mark it complete (low-risk)
  - Write a note in the margin → create a new task or email draft (low-risk: draft is held)
- **Risk-tiered confirmation:** Low-risk actions execute automatically. High-risk actions (decline event, send reply) require explicit confirmation via a follow-up confirmation fax before execution.
- Confirmation fax: Gist sends back a one-page summary of (a) actions auto-executed and (b) high-risk actions awaiting confirmation. User checks/crosses items on the confirmation fax and faxes it back to finalize.
- **Ambiguity definition:** A markup is "ambiguous" if OCR confidence is below 80%, if the markup overlaps multiple content blocks, or if the handwritten text is unrecognizable. Ambiguous markups are listed on the confirmation fax with "?" and require explicit user action.
- **Per-user inbound fax number (provisional):** Each Tier 2 subscriber gets a dedicated Phaxio inbound number. Simpler routing (no cover-page parsing needed). Phaxio number provisioned at subscription start, released at cancellation. At scale (100+ users), revisit shared-number architecture with cover-page routing.
- **Confirmation timeout:** High-risk actions expire after 24 hours if no confirmation fax is received. User receives an email notification at expiry.
- **Inbound fax DPI:** Require fine mode (300 DPI) for inbound faxes to maximize OCR accuracy. OCR testing in success criteria must specify DPI used for each of the 10 test samples.
- **Fax session data model (minimal):** `faxSessions` Firestore collection with fields: `userId`, `outboundFaxId`, `inboundFaxId`, `pendingActions[]`, `status` (awaiting-return | awaiting-confirmation | complete | expired), `createdAt`, `expiresAt`.

### Eureka Insight

Everyone building AI agents assumes they need real-time interactive feedback loops. But for morning planning — where the goal is calm, not speed — a deliberate 1-hour paper loop may produce better decisions than instant screen interactions. The latency is a feature, not a bug. It forces batch processing and deliberate action.

## Design Specifications (added by /plan-design-review)

### Onboarding Flow: Single-Page Form
Single scrollable page, not a multi-step wizard. Feels like filling out a paper form — on screen. Georgia serif headings, system-ui form labels. Background: `var(--bg)` (#fbfbfa). Form container: `.card` pattern.

**Sections (scroll order):**
1. Header: "Set up your morning briefing" (Georgia serif, 22px)
2. Account: email + password fields, or "Sign up with Google" button
3. Connect Calendar: Google Calendar OAuth button ("Connect Google Calendar")
4. Connect Email: Gmail OAuth button ("Connect Gmail")
5. Fax Number: E.164 input with +1 prefix, label: "Where should we fax?"
6. Delivery Time: `<select>` picker, label: "When do you wake up?", sublabel: "We'll fax your Gist 30 minutes before"
7. CTA: "Send my first Gist" (pill button, `var(--green)` accent)
8. Footer: "$8/mo after 7-day trial. Cancel anytime." (muted text, 12px)

### Welcome/Test Fax Design
Branded welcome page using `faxTemplate.ts` patterns. Georgia serif, newspaper header. Content:
- Header: "GIST" masthead (same as daily briefing)
- Subscriber name (personalized)
- "Welcome to Gist. Your first morning briefing arrives tomorrow at [time]."
- Brief description of what to expect (2-3 sentences)
- Footer: mygist.app

NOT a generic test page. This is the first physical touchpoint — it should feel like an invitation.

### PDF Download Button
Placed in Today page footer actions row alongside Print and Resend. Same pill button style (`.button-secondary`). Disabled (grayed, `cursor: not-allowed`) when no gist exists; tooltip: "Your Gist for today hasn't been generated yet."

### Stripe Checkout Return
After Stripe checkout, redirect to `/today` with a success banner: "You're all set! Your first Gist arrives tomorrow at [time]." Banner: `var(--green)` background, auto-dismiss after 5 seconds.

### Failure Notification Email
Subject: "Your morning Gist fax didn't arrive"
Body: One sentence explaining the failure + link to mygist.app/today to view on web. Sent via Resend. Plain, warm, minimal — not a marketing email.

### Accessibility Requirements
- All form inputs: visible labels (not placeholder-as-label), `aria-label` on icon buttons
- OAuth buttons: `role="button"`, `aria-label="Connect Google Calendar"`
- Time picker: native `<select>`, keyboard navigable
- Touch targets: all buttons ≥44px height (increase global button padding from `9px 11px` to `12px 16px`)
- Color contrast: existing palette meets WCAG AA
- PDF download success: `aria-live="polite"` announcement
- Focus management: after OAuth popup closes, return focus to the connection button

### Responsive Behavior
- Onboarding: single column at all viewports. Mobile (<600px): full-width, 16px side padding.
- Today page actions: stack buttons vertically on mobile (full width each).
- Account billing section: already single-column, no changes needed.
- Fax template: fixed US Letter (8.5×11") for print — not responsive (correct behavior).
